## Decision Framework

### The Universal Decision Process

For any decision, use this framework:

```
┌─────────────────────────────────────────────────────────────────┐
│                    1. CLARIFY                                   │
│  What exactly am I deciding? What are the constraints?          │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    2. EXPLORE                                   │
│  What options exist? What has worked before?                   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    3. ANALYZE                                   │
│  What are the tradeoffs? What are the risks?                   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    4. DECIDE                                    │
│  Choose the best option based on analysis                       │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    5. IMPLEMENT                                 │
│  Execute the decision with care                                 │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                    6. EVALUATE                                  │
│  Did it work? What did I learn?                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Decision Checklist

Before making significant decisions:

**Understanding:**
- [ ] Do I understand the problem?
- [ ] Do I understand the context?
- [ ] Do I understand the constraints?
- [ ] Do I understand the implications?

**Options:**
- [ ] Have I considered multiple options?
- [ ] Have I looked for precedents?
- [ ] Have I consulted relevant documentation?
- [ ] Have I sought other perspectives?

**Analysis:**
- [ ] What are the tradeoffs?
- [ ] What are the risks?
- [ ] What are the costs?
- [ ] What are the benefits?

**Validation:**
- [ ] Can I test this decision?
- [ ] How will I know if it's successful?
- [ ] What are the success criteria?
- [ ] What's the rollback plan?

## Meta-Learning Strategies

### Strategy 1: Just-in-Time Learning

**Learn what you need, when you need it, but learn it deeply.**

```
Task Arises
    │
    ├─→ Do I already know how to do this?
    │   ├─ Yes → Apply knowledge
    │   └─ No → Continue
    │
    ├─→ Is this a pattern I recognize?
    │   ├─ Yes → Apply pattern from PATTERN_LANGUAGE.md
    │   └─ No → Continue
    │
    ├─→ Has someone solved this before?
    │   ├─ Yes → Follow their example
    │   └─ No → Continue
    │
    ├─→ Can I investigate to understand it?
    │   └─ Use INVESTIGATION_PROTOCOLS.md
    │
    └─→ Learn the underlying principles
        └─ Add to mental model and documentation
```

### Strategy 2: Pattern Extraction

**Extract patterns from every experience.**

```
Experience
    │
    ├─→ What problem was being solved?
    │
    ├─→ What approach was taken?
    │
    ├─→ Why did it work (or not work)?
    │
    ├─→ What's the general pattern?
    │
    └─→ Where else could this apply?
```

### Strategy 3: Active Experimentation

**Learn by doing, not just reading.**

```
1. Hypothesize
   "I think X will happen if I do Y"

2. Experiment
   "Let me try Y and see what happens"

3. Observe
   "X happened (or Z happened instead)"

4. Analyze
   "Why did X happen? Was my hypothesis correct?"

5. Generalize
   "What's the principle behind X?"

6. Document
   "Record the principle for future use"
```

### Strategy 4: Teaching to Learn

**Solidify understanding by teaching others.**

```
1. Learn something
   "Understand a concept or pattern"

2. Explain it simply
   "Can I explain this in simple terms?"

3. Document it
   "Write clear documentation"

4. Answer questions
   "Respond to others' inquiries"

5. Refine understanding
   "Teaching reveals gaps in understanding"
```

## Practical Application

### Example: Understanding a New Subsystem

**Scenario:** You need to understand the LLM integration subsystem.

**Apply the Knowledge Framework:**

```
1. BEFORE (Preparation)
   ├─ Check META_COGNITION: "What level of understanding do I need?"
   ├─ Check INVESTIGATION_PROTOCOLS: "What's my investigation strategy?"
   └─ Set goals: "Understand how LLM calls are made and handled"

2. SURVEY (Rapid Assessment)
   ├─ Scan llm/ package structure
   ├─ Read LLMClient interface
   ├─ Identify implementations (OpenAI, Groq, Gemini)
   ├─ Note key classes: RequestBuilder, ResponseParser
   └─ Form hypothesis: "Strategy pattern for different providers"

3. INVESTIGATE (Deep Dive)
   ├─ Trace one call: GroqClient.chat()
   │   ├─ How is request built? → RequestBuilder
   │   ├─ How is call made? → HttpClient async
   │   ├─ How is response parsed? → ResponseParser
   │   └─ How are errors handled? → Retry logic
   ├─ Verify hypothesis: "Yes, Strategy pattern"
   ├─ Identify other patterns: Async, Builder, Retry
   └─ Check ARCHITECTURAL_WISDOM: "Why async? Non-blocking"

4. SYNTHESIZE (Extract Knowledge)
   ├─ Patterns used:
   │   ├─ Strategy (LLMClient implementations)
   │   ├─ Builder (LLMRequest)
   │   ├─ Async Request-Response (CompletableFuture)
   │   └─ Retry with Backoff (Resilience4j)
   ├─ Key principles:
   │   ├─ Async by default (don't block game thread)
   │   ├─ Provider-agnostic (easy to switch)
   │   └─ Resilient (retry, circuit breaker)
   └─ Document findings

5. APPLY (Use Knowledge)
   ├─ Can add new LLM provider by implementing LLMClient
   ├─ Can modify request building by extending RequestBuilder
   ├─ Can add custom retry logic
   └─ Can explain design decisions to others
```

### Example: Debugging a Complex Issue

**Scenario:** Agents are getting stuck when pathfinding.

**Apply the Knowledge Framework:**

```
1. DEFINE (Bug Investigation Protocol)
   ├─ What exactly is happening?
   │   "Agent reaches destination but doesn't complete task"
   ├─ When does it happen?
   │   "Only when pathfinding through water"
   ├─ What should happen?
   │   "Agent should complete task after reaching destination"
   └─ Success criteria: "Agent completes water pathfinding tasks"

2. INVESTIGATE (Use Investigation Protocols)
   ├─ Hypothesis 1: "Pathfinding doesn't handle water"
   │   ├─ Test: Agent paths through water → REJECTED
   │   └─ Observation: Agent DOES path through water
   │
   ├─ Hypothesis 2: "Action doesn't detect water completion"
   │   ├─ Test: Add logging to action
   │   ├─ Observation: isComplete() never returns true in water
   │   └─ Test: Check action logic → CONFIRMED
   │
   └─ Root cause: "isComplete() checks only blocks, not water state"

3. DESIGN SOLUTION (Apply Pattern Language)
   ├─ Check patterns: Should we modify State Machine?
   ├─ Check ARCHITECTURAL_WISDOM: Is this a layer boundary issue?
   ├─ Solution: "Add water state to completion check"
   └─ Follow WORK_PATTERNS: "Add test, fix, verify"

4. IMPLEMENT (Use Work Patterns)
   ├─ Write test first (Test Pattern)
   ├─ Modify isComplete() logic
   ├─ Verify with test
   └─ Document the pattern

5. LEARN (Meta-Cognitive Reflection)
   ├─ What pattern caused this?
   │   "Completion detection didn't account for all states"
   ├─ What's the general principle?
   │   "Always check all relevant states, not just default"
   ├─ Where else might this apply?
   │   "Other action completions, other state checks"
   └─ Document: "Add to IMPROVEMENT_OPPORTUNITIES.md"
```

## Conclusion: The Learning Organization

**The Vision:**
